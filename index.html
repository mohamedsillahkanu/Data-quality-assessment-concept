What is Data Quality Assessment (DQA)?

Understanding the four dimensions and why they matter for health data

🎯 Purpose of DQA

Data Quality Assessment (DQA) is a systematic process to evaluate the reliability, accuracy, and completeness of health facility data — particularly routine malaria surveillance data reported into DHIS2. Before any analysis or decision-making, we must verify that the data can actually be trusted. DQA produces a numeric score (0–100) across four dimensions, identifies facilities with poor data quality, and flags specific issues that need correction.

①

Reporting Rate

25 pts

Did facilities actually submit data each month? Measures completeness of reporting over time.

②

Coherency

25 pts

Are the numbers logically consistent? E.g. confirmed cases cannot exceed tested cases.

③

Indicators

25 pts

Are indicator values plausible? Checks for missing values, zeroes, and data type issues.

④

Outliers

25 pts

Are there extreme values far outside the normal distribution for a facility and variable?

Dimension 1: Reporting Rate (Completeness)

Did health facilities submit their data on time and for every expected period?

📖 Concept

Reporting rate measures what fraction of expected data submissions were actually received. A facility that was supposed to report 12 months of data but only submitted 9 months has a reporting rate of 75%. Missing months create gaps in surveillance and undercount disease burden.

Reporting Rate Formula

Reporting Rate = (Months Reported / Expected Months) × 100%
Expected Months = Number of Years × 12  |  Max score contribution: 25 pts

DQA Reporting Score (out of 25)

Scorereporting = (Σ Months Reported / Σ Expected Months) × 25
Calculated across all facilities to produce a single aggregate score for the dataset

Dimension 2: Coherency (Logical Consistency)

Do the numbers make logical sense relative to each other?

📖 Concept

Coherency checks test whether pairs of variables maintain logical relationships. In malaria surveillance, certain rules are medically impossible to violate. For example, you cannot confirm more malaria cases than you tested — that is biologically impossible. Each rule is tested across every facility-month combination. A facility-month that violates a rule is counted as a coherency failure.

Coherency Score Formula

Scorecoherency = (Checks Passed / Total Checks) × 25
Total Checks = Number of rules × Number of facility-month rows | Each row is tested against every coherency rule
📏 The Four Standard Coherency Rules
Rule 1: Confirmed ≤ Tested
Medical Logic
You cannot confirm more malaria cases than you actually tested. Confirmed = RDT/microscopy positive. Tested = total RDTs/microscopy performed. Violation: Confirmed = 120, Tested = 90 → impossible.
Rule 2: Confirmed ≤ Suspected
Clinical Logic
Confirmed cases should be a subset of suspected cases (patients who presented with fever). In some reporting systems tested ≤ suspected also holds. Violation: Confirmed = 80, Suspected = 60.
Rule 3: Tested ≤ Suspected (OPD)
Process Logic
You should only test patients who were first suspected (presented with fever to OPD). Testing more than suspected implies either a data entry error or extra screening being coded differently. Violation: Tested = 200, Suspected = 150.
Rule 4: Treated ≤ Confirmed
Treatment Logic
Standard treatment protocols dictate that only confirmed malaria patients should receive ACT treatment. Treating more than confirmed implies presumptive treatment or data entry issues. Violation: Treated = 110, Confirmed = 95.

Dimension 3: Indicator Consistency

Are the reported values plausible — not missing, not zero when they shouldn't be, not wrong data types?

📖 Concept

Indicator consistency checks verify that each data point for each selected variable is a usable, valid number. Three types of issues are detected: (1) Missing values — the cell is blank or null, (2) Inappropriate values — values that are negative, text strings, or otherwise non-numeric, (3) Unexpected zeros — a facility reported zero for a high-volume indicator in a month it did report (suggesting suppression or error rather than genuine zero disease burden).

Indicator Score Formula

Scoreindicator = ((Total Cells − Issues) / Total Cells) × 25
Total Cells = number of variables × number of facility-month rows | Issues = missing + inappropriate + unexpected zeros
🔍 The Three Issue Types

Missing Values

NULL / blank

Cell has no data at all. Happens when facility submits a form with some fields left empty. Cannot be used in any calculation.

Inappropriate Values

−5, "N/A"

Negative numbers, text strings, or non-numeric entries. Impossible for count variables like confirmed cases or OPD attendance.

Unexpected Zeros

0 when active

Facility reported zero for a month it was active and submitting. For high-volume variables, genuine zeros are rare and suggest suppression or data entry error.

Dimension 4: Outlier Detection

Identifying values that are statistically extreme using the IQR (Interquartile Range) method

📖 Concept: What is an Outlier?

An outlier is a data value that is so far from the typical distribution that it is likely the result of a data entry error, a transcription mistake, or a one-off exceptional event. In DQA we use the IQR (Interquartile Range) method — the same method used by box plots — to define upper and lower bounds. Any value outside those bounds is flagged as an outlier.

📐 The IQR Method — Step by Step
1
Sort all values for a variable and find the quartiles
Take all observed values for a given variable (e.g. "Confirmed Cases") across all facilities and months. Sort them from lowest to highest. Find the 25th percentile (Q1) and the 75th percentile (Q3).
2
Calculate the IQR and the fences
IQR = Q3 − Q1
Lower Fence = Q1 − 1.5 × IQR
Upper Fence = Q3 + 1.5 × IQR
3
Flag any value outside the fences
If a value < Lower Fence OR a value > Upper Fence → it is an outlier. The lower fence is often negative (floored at 0 for count variables). The upper fence is the key threshold in health data.
❌ Low Outlier
Value < Q1 − 1.5×IQR
✅ Normal Range
Q1 − 1.5×IQR  ≤  Value  ≤  Q3 + 1.5×IQR
❌ High Outlier
Value > Q3 + 1.5×IQR
🔢 Worked Example — Confirmed Cases

We have 12 monthly observations of "Confirmed Malaria Cases" for a district. Let's find which are outliers.

1
Observed values (sorted)
Sorted: 12, 18, 22, 25, 28, 31, 34, 38, 42, 48, 55, 210
N = 12 values
2
Find Q1 and Q3
Q1 (25th percentile) = average of 3rd and 4th value = (22 + 25) / 2 = 23.5
Q3 (75th percentile) = average of 9th and 10th value = (42 + 48) / 2 = 45.0
3
Calculate IQR and fences
IQR = Q3 − Q1 = 45.0 − 23.5 = 21.5
Lower Fence = 23.5 − (1.5 × 21.5) = 23.5 − 32.25 = −8.75 → floored at 0
Upper Fence = 45.0 + (1.5 × 21.5) = 45.0 + 32.25 = 77.25
4
Flag outliers
Value 210 > Upper Fence 77.25 → ❌ OUTLIER
All other 11 values are within [0, 77.25] → ✅ Normal
❌
210 confirmed cases in one month is extremely suspicious

The typical range for this district is 12–55 cases. 210 is nearly 4× the upper fence. This could be a data entry error (e.g. 21 entered as 210), an accidental aggregation of multiple months, or a genuine outbreak that should be validated with other sources (e.g. IDSR reports, stock consumption data).

5
Calculate outlier DQA score
Outlier values: 1  |  Total values: 12
Score = ((12 − 1) / 12) × 25 = (11/12) × 25 = 0.917 × 25 = 22.9 / 25

DQA Scoring Methodology

How the four dimension scores combine into a single 0–100 facility score and priority classification

📐 Combined Score Formula

Total DQA Score (Max 100)

DQA Score = Scorereporting + Scorecoherency + Scoreindicator + Scoreoutlier
Each dimension contributes a maximum of 25 points  |  Total maximum = 100 points
🚦 Priority Classification

HIGH Priority

Score < 75

Serious data quality issues. Major problems across multiple dimensions. Facility requires urgent intervention, data cleaning, and retraining.

MEDIUM Priority

75 – 89

Moderate data quality. Some issues present but manageable. Facility needs targeted follow-up and monitoring.

LOW Priority

≥ 90

Good data quality. Minor or no issues detected. Facility data can be used with confidence in analysis.
